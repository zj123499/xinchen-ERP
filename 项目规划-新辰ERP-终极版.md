## 一、系统终极定位

```
旧定位：业务管理系统（CRM）
中间定位：学生生命周期管理 ERP
终极定位：驱动留学公司增长的行业操作系统

核心对象链：
  渠道 → 线索 → 学生 → 合同 → 订单 → 申请 → Offer → 签证 → 入学 → 转介绍
                                    ↑
                              产品/院校/国家资源中心
```

## 三、逐模块详细设计

### 模块 0：基础平台（P0）

```
用户 → 角色 → 权限（四层）
  L1 菜单权限 → 页面/菜单可见
  L2 按钮权限 → 新增/编辑/删除/导出/审批
  L3 数据权限 → 本人/本部门/全部
  L4 字段组权限 → 财务块/学术块/联系方式块 ← 升级为字段组
```

**字段组权限设计（关键升级）**：

```
RoleFieldGroupPermission 表：
  role_id, table_name, field_group, can_read, can_write

field_group 枚举：
  STUDENT.BASIC        — 基础信息
  STUDENT.CONTACT      — 联系方式
  STUDENT.ACADEMIC     — 学术成绩
  STUDENT.FINANCIAL    — 财务信息（合同金额/付款）
  CONTRACT.FINANCIAL   — 合同财务条款
  CONTRACT.LEGAL       — 合同法律条款
  APPLICATION.STATUS   — 申请状态
  ...

为什么是字段组而不是字段？
  - 字段级权限：每次查询 N 次角色匹配，API 灾难
  - 字段组权限：一次查询判定一组，性能提升 10x
  - 实际业务中字段天然成组（财务看财务、学术看学术）
```

### 模块 1：产品资源中心（P0.5 新增）

```
产品资源中心是整个 ERP 的"业务知识库"，所有模块依赖它。

数据模型：
  countries       — 国家（name, code, region, visa_policy JSONB）
  institutions    — 院校（name, country_id, type[大学/学院/中学], ranking, 
                    tuition_range, requirements JSONB）
  majors          — 专业（name, category, institution_id, degree_level,
                    duration, language, tuition, entry_requirements JSONB）
  products        — 产品/服务包（name, business_line_id, country_id, 
                    price, commission_rate, description JSONB）
  product_packages — 产品组合（parent_product_id, child_product_id, discount）

价值：
  - 顾问选校有据可查（查院校→查专业→查录取要求）
  - 合同报价有标准参照（产品→价格→提成率）
  - AI 选校推荐的数据底座
```

### 模块 2：销售 CRM 中心（P2）

#### 2.1 全私海模式 + 防卷三件套（升级）

```
1. 唯一性拦截 + 转介绍申诉
   leads 表：phone/wechat @unique 索引
   撞单流程：
     录入 → 发现已存在 → 提示"该手机号已被[张三]录入" 
     → 若属于转介绍 → 发起申诉(IN_APPEAL)
     → 上传聊天截图 → 主管审批 → 划转/驳回
   
   新增：lead_appeals 表
     id, lead_id, appellant_id, respondent_id, evidence JSONB,
     status[PENDING/APPROVED/REJECTED], approved_by, created_at

2. 行为熔断（风控）
   operation_logs 中监控：
     - 1小时内导出 >50 条线索 → 账号锁定
     - 1天内查看 >30 个学生联系方式 → 预警通知主管
   新增：security_policies 表（可配置规则）

3. 待离职盘点池
   employees 增加 SUSPENDED 状态
   主管操作"待离职盘点" → 该员工名下未签约线索全部进入交接池
   → 主管按规则一键转交 → 完整继承跟进历史
   阻断离职前私下转移客户
```

#### 2.2 渠道归因体系（新增）

```
留学获客渠道复杂：
  搜索引擎(SEO/SEM) → 站群 → 新媒体(抖音/小红书/视频号) → 转介绍 → 合作方

归因模型（可配置）：
  - 首次触达归因：线索来源渠道
  - 末次触达归因：签约前最后渠道
  - 线性归因：所有触点平分
  - 时间衰减：越近触点权重越高

新增表：
  touchpoints    — 触点记录（student_id, channel, source, medium, 
                   campaign, timestamp, url）
  attributions   — 归因结果（touchpoint_id, model, weight, amount）

仪表盘：
  各渠道：花费 → 线索数 → 签约数 → 签约额 → ROI
```

### 模块 4：交付管理中心（P4）

#### 4.1 服务里程碑（已有，保留）

```
service_processes + service_milestones
按业务线+国家动态生成交付节点
```

#### 4.2 审批流引擎（新增，独立于工作流）

```
工作流 = 自动化（触发→条件→动作）
审批流 = 人工决策（提交→逐级审批→通过/驳回）

审批场景：
  合同优惠 >20% → 销售→主管→老板
  退费申请 → 顾问→主管→财务→老板
  报销 >5000 → 员工→主管→财务
  线索划转 → 移交人→接收人→主管

新增表：
  approval_flows    — 审批流程定义
  approval_nodes    — 审批节点（node_type[APPROVE/CC/TRANSFER], approver_role）
  approval_records  — 审批记录（status, comment, attachments, approved_at）

审批特点：
  - 支持会签（全部通过）/ 或签（任一通过）
  - 支持转审（A 转给 B）
  - 支持加签（增加审批人）
  - 支持条件分支（金额 >X 走长流程，≤X 走短流程）
  - 超时自动提醒/自动通过/自动驳回（可配置）
```

### 模块 5：财务利润中心（P5 升级）

#### 5.1 权责发生制 + 留学特化

```
Payment 表升级：
  payment_type: CLIENT_FEE | SCHOOL_COMMISSION | OTHER
  currency: CNY | USD | GBP | AUD | MYR | ...
  exchange_rate: 汇率（用于换算）
  fiscal_year: 财务年度
  fiscal_month: 财务月份 ← 权责发生制核心

为什么需要 fiscal_year/month？
  场景：2026年3月签约，2027年9月入学后学校才返佣
  问题：返佣收入该算在哪个年度？
  答案：按权责发生制，入学时确认收入，记在 2027 年 9 月
  同时：若 2028 年退费，提成需要从 2027 年扣回

Cost 表：
  cost_type: MARKETING | SALARY | COMMISSION | CHANNEL | OPERATION | OTHER
  allocation_rule: 成本分摊规则（按人头/按营收/按面积）

利润模型：
  按维度切分：国家 × 院校 × 顾问 × 渠道 × 月度
  profit = SUM(payments) - SUM(costs)
  预计算 + 实时查询混合
```

#### 5.2 分段提成释放引擎（升级）

```
CommissionRule 表增加版本控制：
  version, effective_start, effective_end ← 历史规则保留

提成释放节点：
  1. 首款到账 → 签约顾问 30%
  2. 尾款到账 → 签约顾问 30%
  3. Offer 到达 → 规划顾问 20% + 文书 20%
  4. 入学确认 → 签约顾问 40%
  5. 返佣到账 → 签约顾问 100% 返佣提成

退费扣回：
  退费审批通过 → 按已释放比例扣回各角色提成
  → 生成负向 commission 记录
  → 从下月薪资中扣除
```

### 模块 6：客户成功中心（P6 新增）

```
留学行业的转介绍是最大增长引擎。

customer_success 体系：
  satisfaction_surveys   — 满意度调查（NPS, 各环节评分）

    奖励类型：现金/优惠券/服务升级
```

### 模块 7：风险管理中心（P7 新增）

```
风险规则引擎（主动发现，不是被动记录）：

risk_rules 表：
  规则示例：
    1. 距离申请截止 < 7天 AND 材料完成率 < 80% → 高风险
    2. 签证递交 > 30天 AND 无结果 → 中风险
    3. 付款逾期 > 15天 → 中风险
    4. 学生投诉 > 2次 → 高风险
    5. 顾问名下休眠线索 > 50 → 中风险

risk_records：
  student_id, risk_rule_id, risk_level, status, detected_at, resolved_at

risk_notifications：
  风险推送：钉钉/邮件/系统消息

老板每日风险看板：
  高风险学生 8 人：
    3 人 材料未完成（截止临近）
    2 人 签证延期
    2 人 付款异常
    1 人 投诉未处理
```

### 模块 8：AI 智能层（P8 新增）

```
AI 不是替代人工，是增强顾问能力。

架构：
  ERP 业务层
      ↓
  AI Gateway（统一入口，管理 API Key/模型选择/限流/缓存）
      ↓
  AI 模型（GPT-4/Claude/文心/通义，可切换）

三个 AI Agent：

1. AI 选校顾问
   输入：学生 GPA/语言成绩/预算/专业偏好
   查询：院校数据库 + 历史录取案例
   输出：冲/稳/保 三档院校推荐 + 申请策略
   价值：新顾问也能给出专业选校方案

2. AI 文书助手
   输入：学生经历/目标专业/院校要求
   辅助：PS 大纲/推荐信框架/简历优化
   不替代文书老师，而是提供初稿和灵感

3. AI 客服助手
   输入：FAQ 库 + 院校信息 + 政策信息
   回答：家长常见问题（费用/流程/签证/住宿）
   人工接管：复杂问题自动转人工

4. AI 经营分析（未来）
   自然语言查询："上个月签约最多的是哪个顾问？"
   AI 翻译为 SQL → 执行 → 返回结果 + 图表
```

### 模块 9：配置版本管理（基础平台增强）

```
所有配置表增加版本控制：

commission_rules:
  version, effective_start, effective_end

workflows:
  workflow_version, published_at, published_by

business_lines.form_schema:
  每次修改保留历史 JSON

价值：
  - 2026年提成10%，2027年改为8%，历史订单仍按10%算
  - 工作流误改可回滚
  - 合规审计：谁、什么时候、改了什么
```

### 模块 10：审计中心（基础平台增强）

```
audit_logs 表：
  id, table_name, record_id, action[CREATE/UPDATE/DELETE],
  field_name, old_value, new_value, operator_id, operator_ip,
  operated_at

示例：
  员工：张三
  时间：2026-07-15 14:30
  操作：修改学生手机号
  表：students, record_id=1024
  字段：phone
  旧值：138xxxx1234
  新值：139xxxx5678

审计日志 vs 操作日志：
  操作日志：谁做了什么（登录、导出、删除）
  审计日志：什么数据被改了，改之前是什么，改之后是什么
  两者互补
```

## 七、关键设计决策

### 1. 权限：字段组而非字段
```
字段级权限在 Prisma 架构下是性能灾难。
字段组权限（FINANCIAL/ACADEMIC/CONTACT）性能提升 10x，业务隔离效果相同。
```

### 2. 线索：全私海 + 申诉机制
```
不做自动回收（公海模式），而是：
  撞单 → 拦截 + 申诉 → 主管裁决
  休眠 → 标红预警 → 主管评估 → 手动切单
  离职 → 待离职盘点池 → 一键交接
```

### 3. 财务：权责发生制
```
留学行业长周期特性决定：
  必须有 fiscal_year/month，否则跨年退费/返佣无法正确核算。
```

### 4. 审批流 vs 工作流
```
工作流 = 自动化（系统自己跑）
审批流 = 人工决策（人审批）

两者独立设计，不混在一起。
```

### 5. 配置版本管理
```
所有可配置项（提成规则/工作流/表单Schema）保留历史版本。
历史订单按当时规则计算，不受未来规则变更影响。
```

### 6. AI 作为增强层
```
AI 不替代人工，而是增强顾问能力。
AI Gateway 统一管理，模型可切换，成本可控。
```

### 7. 审计中心
```
操作日志 + 审计日志 双轨制：
  操作日志：谁在什么时候做了什么操作
  审计日志：什么数据被改了，改之前是什么，改之后是什么
```
