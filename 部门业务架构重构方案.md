# 新辰ERP — 部门业务架构重构方案

> 版本：v3.2  
> 定位：从「表单驱动」升级为「部门业务驱动」  
> 核心理念：按留学公司实际部门划分 → 设计业务流转 → 定义表单字段 → 配置权限矩阵  
> 更新记录：  
> - v3.1 — 市场部分立新媒体部、网络部；客户成功部改为老客户跟进
> - v3.2 — 重构核心业务流：多部门录入线索→分配顾问→客户管理→签约触发交付

---

## 一、部门架构全景

```
                         ┌──────────────────────────────────────────┐
                         │           总经办 / 管理层                  │
                         │     经营驾驶舱 / BI / 风险监控            │

                                           │
        ┌───────┬───────┬───────┬─────────┼─────────┬───────┬───────┬───────┬───────┐
        │       │       │       │         │         │       │       │       │       │
      市场部  新媒体部  网络部  销售部    交付部    财务部   人事部   老客户   产品    管理层
      (获客)  (矩阵)  (SEO)  (转化)    (服务)    (核算)   (组织)   跟进   资源部   (决策)
                                                                 (售后)  (知识库)
```

### 十大部门定义

| 部门 | code | 核心职责 | 关键岗位 |
|------|------|----------|----------|
| **市场部** | `marketing` | 渠道获客、SEM/信息流投放、合作方管理、品牌推广 | 市场总监、市场专员 |
| **新媒体部** | `new_media` | 抖音/小红书/公众号/视频号等新媒体矩阵运营 | 新媒体主管、新媒体运营 |
| **网络部** | `network` | 站群管理、SEO优化、网站运营维护 | 网络运营专员 |
| **销售部** | `sales` | 线索跟进、客户咨询、需求分析、签约转化 | 销售总监、顾问 |
| **交付部** | `delivery` | 申请递交、文书写作、Offer跟进、签证办理、入学指导 | 交付总监、文书顾问、申请顾问 |
| **财务部** | `finance` | 收款付款、成本核算、提成计算、薪资核算、利润分析 | 财务总监、会计 |
| **人事行政部** | `hr` | 员工管理、组织架构、考勤、权限配置、行政事务 | 人事总监、HR专员 |
| **老客户跟进** | `old_customer` | 回访关怀、满意度调查、投诉处理、转介绍运营 | 客户专员 |
| **产品资源部** | `product` | 院校专业维护、产品定价、知识库管理、竞品分析 | 产品总监、产品经理 |
| **管理层** | `management` | 经营看板、数据大屏、风险监控、战略决策 | 总经理、副总经理、区域总监 |

## 三、状态机设计

### 3.1 学生生命周期状态机（核心流）

```
                      四部门均可录入线索
     ┌──────────────────────┐
     │  市场/新媒体/网络/销售  │
     └──────────┬───────────┘
                │
                ▼
         ┌──────────┐
         │  NEW     │  新线索（未分配）
         │  新建     │
         └────┬─────┘
              │ 分配顾问（assign）
              ▼
         ┌──────────┐
         │ ASSIGNED │  已分配（进入顾问客户管理）
         │ 已分配    │
         └────┬─────┘
              │ 顾问开始跟进
              ▼
         ┌──────────┐          ┌──────────┐
    ┌───→│ CONSULTING│─────────→│  LOST    │  流失
    │    │ 咨询中    │          │  已流失   │
    │    └────┬─────┘          └──────────┘
    │         │
    │         │ 修改状态为「已签约」★ 核心触发点
    │         ▼
    │    ┌──────────┐
    │    │ SIGNED   │  已签约 → 自动推送交付部
    │    │ 已签约    │
    │    └────┬─────┘
    │         │
    │         ▼ 交付部-文书老师处理
    │    ┌──────────┐
    │    │DELIVERING│  交付中
    │    │ 交付中   │
    │    └────┬─────┘
    │         │
    │         ▼
    │    ┌──────────┐
    │    │COMPLETED │  已完成
    │    │ 已完成    │
    │    └────┬─────┘
    │         │
    │         ▼
    │    ┌──────────┐          ┌──────────┐
    └────│ REFERRAL │─────────→│ 回到线索池 │
         │ 转介绍    │          │ (重新录入)  │
         └──────────┘          └──────────┘
```

**核心流转规则：**

| 当前状态 | 可转换状态 | 触发者 | 说明 |
|----------|-----------|--------|------|
| NEW | ASSIGNED | 市场/新媒体/网络/销售部门管理者 | 分配顾问，学生信息进入顾问客户管理 |
| NEW | LOST | 任意 | 无效线索直接关闭 |
| ASSIGNED | CONSULTING | 顾问 | 顾问开始跟进 |
| ASSIGNED | LOST | 顾问 | 确认无法跟进 |
| CONSULTING | SIGNED | 顾问 | **核心节点**：修改为已签约，自动触发推送交付部 |
| CONSULTING | LOST | 顾问 | 跟进失败 |
| SIGNED | DELIVERING | 系统自动 → 交付部文书老师 | 学生详细信息推送到交付部 |
| DELIVERING | COMPLETED | 交付部 | 入学完成 |
| COMPLETED | REFERRAL | 老客户跟进 | 触发转介绍 |

## 五、权限矩阵设计

### 5.1 四维权限模型

```
权限体系 = 部门(Department) × 岗位(Position) × 角色(Role) × 数据范围(DataScope)

第一维 — 部门：决定「可见哪些业务模块」
第二维 — 岗位：决定「能做哪些操作」
第三维 — 角色：决定「菜单+操作+字段」权限组合
第四维 — 数据范围：决定「看谁的/哪个部门的数据」
```

### 5.2 16个核心角色

| 角色 | code | 所属部门 | 菜单范围 | 数据范围 | 核心权限 |
|------|------|----------|----------|----------|----------|
| **超级管理员** | `admin` | - | 全部 | 全部 | 系统所有权限 |
| **市场总监** | `marketing_director` | 市场部 | 市场+线索池+BI | 全部 | 渠道管理、线索录入、分配顾问、ROI查看 |
| **市场专员** | `marketing_specialist` | 市场部 | 市场+线索录入 | 本人 | 渠道创建、触点管理、录入线索 |
| **新媒体主管** | `media_director` | 新媒体部 | 新媒体+线索池 | 本部门 | 账号管理、线索录入、分配顾问 |
| **新媒体运营** | `media_operator` | 新媒体部 | 新媒体+线索录入 | 本人 | 账号运营、数据录入、录入线索 |

## 七、实施路线图

### Phase 1: 数据层重构（2-3天）
- 升级 Prisma Schema：增加 department 字段到各业务表
- 设计业务流转状态机常量
- 更新种子数据：16个角色 + 权限码体系
- 创建权限矩阵工具函数

### Phase 2: API 层重构（3-4天）
- 按部门分组 API 路由
- 实现统一的部门权限校验中间件
- 实现字段组权限过滤
- 实现跨部门数据流转 API

### Phase 3: 前端重构（4-5天）
- 新导航系统按部门分组
- 工作台(各部门角色不同看到不同内容)
- 各部门核心表单页面
- 学生360°视图聚合页

### Phase 4: 业务流程闭环（3-4天）
- ★ **核心流程**: 四部门录入线索 → 分配顾问 → 顾问客户管理 → 已签约 → 自动推送交付部
- 分配顾问时：学生信息从线索池复制到顾问客户管理视图
- 签约触发：自动创建交付任务、通知财务生成应收、更新渠道ROI
- 交付部 → 文书老师处理 → 状态变更自动通知老客户跟进
- 财务部 → 收款关联提成自动计算

### Phase 5: 测试与部署（1-2天）
- 权限回归测试
- 业务流程端到端测试
- Docker 部署更新

## 八、新旧架构对比

| 维度 | 旧架构 | 新架构 |
|------|--------|--------|
| **组织方式** | 按数据表/表单归类 | 按部门业务归类 |
| **导航结构** | 9个扁平模块 | 10个部门 + 5个系统模块 |
| **表单设计** | 通用CRUD表单 | 按业务场景定制的上下文表单 |
| **业务流转** | 隐式的数据关联 | 显式的状态机 + 跨部门流转 |
| **权限体系** | 角色-权限 一对多 | 部门+岗位+角色+数据范围 四维 |
| **角色数量** | 5个 | 16个（覆盖各部门岗位） |
| **权限粒度** | 菜单+操作 | 菜单+操作+字段组+数据范围 |
