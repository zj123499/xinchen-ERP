# 新辰ERP 开发原则 v1.0

> 适用范围：部门业务架构重构项目（v3.1）  
> 制定依据：现有代码模式 + 行业最佳实践  
> 约束等级：🔴 强制 → 🟡 推荐 → 🟢 可选

---

## 一、架构原则

### 1.1 分层架构 🔴

```
┌─────────────────────────────────────────────┐
│  页面层 (page.tsx)                           │  ← 组装 UI，编排用户交互
├─────────────────────────────────────────────┤
│  业务组件层 (components/)                    │  ← 可复用 UI 单元
├─────────────────────────────────────────────┤
│  API 层 (api/**/route.ts)                    │  ← 数据验证、权限守卫、业务编排
├─────────────────────────────────────────────┤
│  服务层 (lib/services/)  【新增】             │  ← 核心业务逻辑、跨模块复用
├─────────────────────────────────────────────┤
│  数据层 (Prisma)                             │  ← 数据库访问
└─────────────────────────────────────────────┘
```

**规则：**
- 页面层 **不得** 直接调用 Prisma Client
- API 层 **不得** 包含复杂业务逻辑（> 30 行），必须抽取到服务层
- 服务层函数签名统一：`(ctx: AuthContext, ...args) => Promise<Result<T>>`

### 1.2 模块化边界 🔴

每个部门模块独立为目录：

```
src/
├── app/
│   └── (main)/
│       ├── marketing/        # 市场部
│       ├── new-media/        # 新媒体部
│       ├── network/          # 网络部
│       ├── sales/            # 销售部
│       ├── delivery/         # 交付部
│       ├── finance/          # 财务部
│       ├── hr/               # 人事部
│       ├── old-customer/     # 老客户跟进
│       ├── product/          # 产品资源部
│       └── bi/               # 经营驾驶舱
├── api/
│   ├── marketing/            # 市场部 API
│   ├── new-media/            # 新媒体部 API
│   ├── network/              # 网络部 API
│   ├── sales/                # 销售部 API
│   ...
├── lib/
│   ├── services/
│   │   ├── marketing/        # 市场部服务
│   │   ├── sales/            # 销售部服务
│   │   └── ...
│   ├── auth.ts
│   ├── permission.ts
│   └── ...
└── components/
    ├── ui/                   # 通用 UI 组件
    ├── marketing/            # 市场部专用组件
    └── ...
```

**模块隔离规则：**
- 模块间只能通过 API 路由或服务层函数通信
- 禁止跨模块直接引用 Prisma 模型
- 禁止页面 A 直接 import 页面 B 的组件

### 1.3 单一职责 🔴

- 每个文件只做一件事
- 函数不超过 80 行（超长必须拆分）
- 组件不超过 300 行（超长必须抽取子组件）

## 二、TypeScript 规范

### 2.1 严格模式 🔴

项目已启用 `"strict": true`，以下规则必须遵守：

| 规则 | 要求 |
|------|------|
| `strictNullChecks` | 必须处理 null/undefined |
| `noImplicitAny` | 禁止隐式 any |
| `noImplicitThis` | 禁止隐式 this |
| `strictFunctionTypes` | 函数类型参数逆变检查 |

### 2.2 类型定义规范 🔴

```typescript
// ✅ 正确：明确的类型定义

// ❌ 错误：使用 any
```

状态、类型等固定值统一使用枚举或 const 对象：

## 三、数据库原则

### 3.1 Prisma Schema 规范 🔴

- 所有表必须有 `createdAt`、`updatedAt`（use `@default(now())` 和 `@updatedAt`）
- 主键统一 `id: String @id @default(cuid())`
- 外键命名：`{关联表名}Id`
- 所有业务表必须有 `tenantId: String` 字段（多租户隔离）
- 部门字段统一：`department: String?`（指向部门 code）
- 使用 `@@index` 为频繁查询字段建索引
- 关系字段使用 `relation` 双向定义

```prisma
model Lead {
  id              String     @id @default(cuid())
  tenantId        String
  department      String?    // 'marketing' | 'sales'
  name            String
  phone           String
  status          String     @default("new")
  source          String?
  consultantId    String?
  consultant      Employee?  @relation(fields: [consultantId], references: [id])
  
  createdAt       DateTime   @default(now())
  updatedAt       DateTime   @updatedAt
  
  @@index([tenantId, status])
  @@index([tenantId, phone])
  @@index([tenantId, consultantId])
}
```

### 3.2 查询规范 🔴

```typescript
// ✅ 正确：select 按需加载，不查全表
const lead = await prisma.lead.findUnique({
  where: { id },
  select: {
    id: true,
    name: true,
    status: true,
    consultant: { select: { id: true, name: true } },
  },
});

// ✅ 正确：分页查询
const [list, total] = await Promise.all([
  prisma.lead.findMany({
    where,
    skip: (page - 1) * pageSize,
    take: pageSize,
    orderBy: { createdAt: 'desc' },
    select: listSelect,
  }),
  prisma.lead.count({ where }),
]);

// ❌ 错误：全表 include
const lead = await prisma.lead.findUnique({
  where: { id },
  include: { consultant: true, student: true, orders: true, ... },
});
```

### 3.3 事务规范 🟡

涉及多表写操作必须使用交互式事务：

```typescript
await prisma.$transaction(async (tx) => {
  const lead = await tx.lead.update({ ... });
  await tx.student.create({ ... });
  await tx.operationLog.create({ ... });
});
```

## 四、API 设计原则

### 4.1 路由命名规范 🔴

```
GET    /api/{module}              → 列表
POST   /api/{module}              → 创建
GET    /api/{module}/[id]         → 详情
PUT    /api/{module}/[id]         → 更新
DELETE /api/{module}/[id]         → 删除
POST   /api/{module}/[id]/{action} → 业务动作（状态变更等）
```

**业务动作命名**：`assign`、`transfer`、`approve`、`reject`、`appeal`、`restore`

### 4.2 统一响应格式 🔴

```typescript
// 分页列表
{ total: 100, page: 1, pageSize: 20, totalPages: 5, list: [...] }

// 单条数据
{ ...data }

// 操作结果
{ success: true, message: "操作成功" }

// 错误
{ error: "错误描述" }

// 业务冲突（如手机号重复）
{ error: "手机号已存在", conflict: { type: "phone_duplicate", leadId: "xxx" } }
```

### 4.3 权限守卫 🔴

**每个 API 路由必须进行权限检查**：

```typescript
import { getAuthContext, requirePermission } from '@/lib/permission';

export async function GET(request: NextRequest) {
  const ctx = getAuthContext(request);
  
  // 方式1：要求特定权限
  requirePermission(ctx, 'sales:leads:view');
  
  // 方式2：在服务层注入权限上下文
  const result = await leadService.list(ctx, params);
  // ...
}
```

### 4.4 错误处理规范 🔴

```typescript
export async function POST(request: NextRequest) {
  try {
    const ctx = getAuthContext(request);
    requirePermission(ctx, 'sales:leads:create');
    
    const body = await request.json();
    
    // 1. 参数校验
    if (!body.name || !body.phone) {
      return NextResponse.json(
        { error: '姓名和手机号为必填项' },
        { status: 400 }
      );
    }
    
    // 2. 业务校验
    const exists = await prisma.lead.findFirst({
      where: { tenantId: ctx.tenantId, phone: body.phone, status: { not: 'lost' } },
    });
    if (exists) {
      return NextResponse.json(
        { error: '该手机号已有活跃线索', conflict: { type: 'phone_duplicate', leadId: exists.id } },
        { status: 409 }
      );
    }
    
    // 3. 执行操作
    const lead = await leadService.create(ctx, body);
    
    // 4. 记录操作日志
    await logOperation(ctx, 'lead:create', lead.id);
    
    return NextResponse.json(lead, { status: 201 });
  } catch (error) {
    console.error('POST /api/leads error:', error);
    return NextResponse.json(
      { error: '服务器内部错误' },
      { status: 500 }
    );
  }
}
```

### 4.5 分页规范 🔴

- 默认 `pageSize = 20`，最大 `100`
- 搜索关键词：`keyword` 参数
- 排序参数：`sortBy` + `sortOrder`
- 筛选参数：平铺在 query string 中

- 页面组件：`"use client"` + 服务端数据获取架构（如需要）
- 组件文件命名：`PascalCase`（如 `LeadTable.tsx`）
- 页面文件命名：`page.tsx`（Next.js 约定）
- 布局文件命名：`layout.tsx`（Next.js 约定）

### 5.2 样式规范 🔴

- **纯 Tailwind CSS**，不使用 CSS Module 或 styled-components
- 提取长 class 为组件或使用 `cn()` 工具函数
- 颜色：使用 Tailwind 内置色系（`blue-600`、`green-500` 等）
- 间距：使用 Tailwind 内置间距系统（`p-4`、`gap-6` 等）

```typescript

// 条件样式

  'px-4 py-2 rounded-lg text-sm font-medium',

```

### 5.3 数据获取规范 🔴

- 页面内使用原生 `fetch()` 调用 API
- 使用 `useCallback` 包装数据获取函数
- Loading、Empty、Error 三种状态必须有 UI 处理

```typescript
const [data, setData] = useState<Lead[]>([]);
const [loading, setLoading] = useState(true);
const [error, setError] = useState<string | null>(null);

const fetchData = useCallback(async () => {
  setLoading(true);
  setError(null);
  try {
    const res = await fetch(`/api/leads?${params}`);
    if (!res.ok) throw new Error('请求失败');
    const json = await res.json();
    setData(json.list);
  } catch (e) {
    setError(e instanceof Error ? e.message : '未知错误');
  } finally {
    setLoading(false);
  }
}, [params]);
```

### 5.4 表单处理规范 🔴

```typescript
// 创建/编辑表单必须处理：
// 1. 必填校验  2. 格式校验  3. 唯一性校验  4. 提交中状态  5. 错误提示

const handleSubmit = async (e: React.FormEvent) => {
  e.preventDefault();
  
  // 客户端校验
  if (!form.name.trim()) {
    setFieldError('name', '姓名为必填项');
    return;
  }
  
  setSubmitting(true);
  try {
    const res = await fetch('/api/leads', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(form),
    });
    
    if (res.status === 409) {
      const { conflict } = await res.json();
      setFieldError('phone', '该手机号已存在');
      return;
    }
    
    if (!res.ok) throw new Error('提交失败');
    
    // 成功 → 关闭弹窗、刷新列表
    onSuccess();
  } catch (e) {
    setSubmitError('提交失败，请重试');
  } finally {
    setSubmitting(false);
  }

```

## 六、权限体系原则

### 6.1 权限码编码规范 🔴

```
{department}:{resource}:{action}

示例:
  sales:leads:create        → 销售部-线索-创建
  sales:leads:assign        → 销售部-线索-分配
  finance:receivables:view  → 财务部-应收-查看
  delivery:applications:assign → 交付部-申请-分配
```

### 6.2 四位权限模型 🔴

每次权限判断检查四个维度：

| 维度 | 来源 | 说明 |
|------|------|------|
| **菜单** | 角色→菜单树 | 能否看到此页面/功能入口 |
| **操作** | 角色→权限码 | 能否执行此操作（CRUD+业务动作） |
| **字段** | 角色→字段组 | 能看到/编辑哪些字段 |
| **数据** | 部门+岗位 | 能访问哪些数据范围（本人/本部门/全部） |

### 6.3 数据权限实现 🔴

```typescript
// 在服务层构建 where 条件时注入数据范围
function applyDataScope(ctx: AuthContext, baseWhere: any) {
  if (isAdmin(ctx)) return baseWhere; // 管理员看全部
  
  const deptCodes = getDeptCodesFromRoles(ctx.roles);
  
  if (isDeptDirector(ctx)) {
    // 部门总监：看本部门
    return { ...baseWhere, department: { in: deptCodes } };
  }
  
  // 普通员工：只看本人
  return { ...baseWhere, OR: [
    { consultantId: ctx.userId },
    { createdBy: ctx.userId },
  ]};
}
```

## 七、业务流转原则

### 7.1 状态机定义 🔴

每个业务状态机明确定义：
- 所有状态枚举
- 合法状态转换映射
- 每个转换的前置条件
- 每个转换的副作用（通知、日志、数据联动）

```typescript
// 状态机定义示例
const STATE_MACHINE = {
  states: ['new', 'in_progress', 'completed', 'cancelled'],
  transitions: {
    new: {
      to: ['in_progress', 'cancelled'],
      conditions: { in_progress: (ctx) => ctx.role === '...' },
      effects: { in_progress: (ctx) => notifyAssignee(ctx) },
    },
    // ...
  },
};
```

### 7.2 跨部门事件 🔴

跨部门流转通过事件驱动，不得在业务代码中直接操作其他部门的数据：

```typescript
// ✅ 正确：通过事件总线
await eventBus.emit('lead:signed', { leadId, studentId, contractId });
// → 市场部更新渠道业绩
// → 财务部创建应收记录
// → 交付部创建交付任务

// ❌ 错误：直接操作
await prisma.deliveryTask.create({ ... });  // 销售部代码里直接创建交付任务
```

### 7.3 事件实现 🔴

使用服务层函数调用替代完整事件总线（避免过度工程）：

```typescript
// 签约后联动
async function handleLeadSigned(ctx: AuthContext, leadId: string) {
  const lead = await prisma.lead.findUnique({ where: { id: leadId } });
  
  // 串行执行副作用（失败不影响主流程）
  await Promise.allSettled([
    updateChannelPerformance(ctx, lead.source),
    createDeliveryTask(ctx, leadId),
    notifyFinanceForReceivable(ctx, leadId),
    logOperation(ctx, 'lead:signed', leadId),
  ]);
}
```

## 八、代码质量原则

### 8.1 命名规范 🔴

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | 小写+kebab-case | `lead-service.ts`, `use-leads.ts` |
| 组件文件 | PascalCase | `LeadTable.tsx`, `SearchBar.tsx` |
| 函数 | camelCase | `getLeadById()`, `handleSubmit()` |
| 接口 | PascalCase, I无前缀 | `LeadCreateInput`, `ApiResponse<T>` |
| 类型别名 | PascalCase | `LeadStatus`, `DepartmentCode` |
| 枚举 | PascalCase | `LeadStatus`, `Department` |
| 常量 | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE`, `DEFAULT_ROLES` |
| Prisma 模型 | PascalCase | `Lead`, `Contract` |
| 数据库字段 | camelCase | `createdAt`, `tenantId` |
| API 路由 | kebab-case | `/api/sales/leads/transfers` |

### 8.2 注释规范 🟡

```typescript
/**
 * 服务层函数需要 JSDoc 注释
 * @param ctx - 认证上下文（userId, tenantId, roles）
 * @param input - 创建参数
 * @returns 创建的线索对象（含关联顾问信息）
 * @throws {PermissionError} 无权限时抛出
 */
async function createLead(ctx: AuthContext, input: LeadCreateInput): Promise<Lead> {
  // 关键业务逻辑加单行注释说明意图
  // 非显著操作必须注释（如：兜底逻辑、特殊边界处理）
}
```

### 8.3 错误处理原则 🔴

```
用户操作 → 客户端校验 → API 参数校验 → 权限校验 → 业务校验 → 执行 → 记录日志
   ↓            ↓             ↓            ↓          ↓         ↓        ↓
 友好提示   fieldError     400+msg       403      409+conflict  201    silent
```

- **绝不** 让用户看到技术错误栈
- **绝不** 静默吞异常（至少 `console.error` + 日志）
- **必须** 区分业务异常（4xx）和系统异常（500）
- **推荐** 业务异常携带 `conflict` 信息方便前端处理

### 8.4 日志规范 🔴

```typescript
// 操作日志：用户做了什么
await logOperation(ctx, 'lead:create', leadId, {
  before: null,
  after: { name: input.name, phone: input.phone },
});

// 登录日志：谁在什么时候登录
await logLogin(ctx.userId, request, success);

// 错误日志（仅服务端 console）
console.error('Failed to create lead:', { userId: ctx.userId, error });
```

## 十、测试原则

### 10.1 测试层次 🟡

```
优先级（从高到低）:
1. 关键业务逻辑单元测试（状态机、权限计算）
2. API 路由集成测试（权限守卫 + 核心 CRUD）
3. 关键流程 E2E 测试（线索→签约→交付）
```

### 10.2 测试文件组织 🟡


## 十一、重构执行原则

### 11.1 渐进式迁移 🔴

- **不改数据库结构** —— 在现有表基础上增加 `department` 字段，不破坏已有数据
- **不改中间件** —— 现有 JWT 认证机制保持不变
- **并行运行** —— 新旧路由可同时存在，通过菜单配置切换
- **先建后拆** —— 新模块建好后，运行验证通过后再删除旧代码

### 11.2 每个 Phase 的验收标准 🔴

| Phase | 验收标准 |
|-------|----------|
| Phase 1 数据层 | Prisma migrate 无报错，种子数据可正常初始化 |
| Phase 2 API | 所有新增 API 返回正确格式，权限守卫生效 |
| Phase 3 前端 | 所有页面可正常渲染，菜单按部门正确分组 |
| Phase 4 业务流程 | 核心流转链路（线索→签约→交付→回访）端到端可走通 |
| Phase 5 部署 | 生产环境运行稳定 24 小时无严重错误 |

### 11.3 回滚策略 🔴

- 数据库 migration 始终可回滚
- 每次部署前打 tag
- 保留旧路由至少一个版本周期
