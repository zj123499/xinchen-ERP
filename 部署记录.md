## 一、部署流水线说明

协作约定：本地改代码，攒一批后一次性上传部署。每批完成后追加记录并运行 `deploy_batch.js` 上传到服务器。

标准流程：
1. 本地改码 → `deploy_batch.js` 打包上传到服务器
2. `docker compose build --no-cache` → `docker compose up -d`
3. 容器 entrypoint 自动执行 `prisma db push`（建表/迁移）+ `prisma db seed`（播种）
4. 验证：`health` 健康检查、关键 API 返回 200

技术栈：Next.js 14（App Router, standalone 构建）+ TypeScript + Prisma + PostgreSQL（腾讯云实例 `1Panel-postgresql-Bq8Z`，库 `xinchen_erp`）。多租户（`tenantId`）+ JWT 鉴权（中间件注入 `x-user-id`/`x-user-roles`，dashboard 直接从 JWT 解析 `tenantId`/`userId`/`roles`）。

## 三、功能修复记录

### 修复一：账号和密码设置逻辑优化（避免误解）— 提交 `51ebed1`
**问题**：原「员工管理」创建/编辑账号时，密码输入框语义不清——留空=默认密码、填写=自定义密码，但界面无任何提示，管理员易误以为「填写了却没生效」或「忘记改密码员工一直用默认密码登录」，造成沟通与安全隐患。

**根因**：
- 后端 `password` 留空即静默使用默认密码 `Xc@123456`，无显式标识
- 前端只有单个密码框，无「确认密码」「显示密码」「当前状态」提示
- 列表页无法区分「默认密码 / 自定义密码 / 待改密」

**改动清单**：

| 文件 | 改动 |
|------|------|
| `prisma/schema.prisma` | `User` 新增 `isDefaultPassword Boolean @default(true)`（映射 `is_default_password`） |
| `src/app/api/employees/route.ts` | POST 解析 `confirmPassword`，`password !== confirmPassword` 返回 400；创建时 `isDefaultPassword: !password` |
| `src/app/api/employees/[id]/route.ts` | PUT 同样校验两次密码一致；重置/编辑维护 `isDefaultPassword` |
| `src/app/(main)/employees/page.tsx` | 新增「显示密码」开关、二次「确认密码」输入、动态提示文案；列表「登录账号」列新增徽标：默认密码=琥珀、自定义=绿、待改密=蓝 |
| `prisma/seed.ts` | admin 与管理员示例账号标记 `isDefaultPassword: false` |
| `src/app/api/auth/change-password/route.ts` | 自助改密成功后置 `isDefaultPassword: false` |

**验证（服务器）**：自定义密码创建 → 登录 200 ✅；两次密码不一致 → 创建拦截 400 ✅；重置为默认密码 → 返回 `isDefaultPassword: true` 200 ✅

### 修复二：工作台按个人权限展示（不再全员看全公司）— 提交 `3d7694e`
**问题**：工作台（`/`）原本对所有登录人都展示全公司汇总数据（线索总量、合同总额、回款等），与「按权限隔离数据」设计冲突，存在信息越权风险。

**根因（排查链路）**：
1. **残留 docker build 冲突**：上一次 `docker compose build` 未结束，新构建被 CANCELED，部署停留在旧镜像 → `pkill -9 -f "docker compose build"` 后重部署
2. **错误导入鉴权函数**：曾误从 `@/lib/auth` 导入 `verifyToken/getTokenFromRequest`（该文件仅导出 `hashPassword/verifyPassword`），引发 TS 类型错误 → 移除该依赖
3. **中间件注入的 `x-tenant-id` 头恒为 0**（关键）：dashboard 依赖中间件注入头，实测 `x-user-id=1`、`x-user-roles=admin` 正常但 `tenantId=0`，所有 `where: { tenantId: 0 }` 计数恒为 0 → 改为**直接从 JWT payload 解析** `tenantId/userId/roles`
4. **Dockerfile 吞错**：原 `RUN npx next build 2>&1 | tail -20` 把构建失败错误码吞掉，误判成功并停留旧版 → 改为 `RUN npx next build`，注释标注「不要用 `| tail` 吞掉错误码」

**最终方案**：`src/app/api/dashboard/route.ts` 改为从 `@/lib/jwt` 的 `verifyToken` 解析 `tenantId/userId/roles`；`ALL_SCOPE_ROLES = ["admin","manager","finance"]`，命中则 `scope="all"`，否则 `scope="self"`；按数据归属过滤（Lead/Student/Order→`assignedToId`；Contract/Payment/Application→`student.assignedToId`；Commission→`employee.userId`；VisitRecord→`visitorId`；成本仅全量角色可见）；响应体新增 `scope`。前端 `page.tsx` 标题区加徽标（`我的工作台（仅看我负责的数据）`/`全公司看板`），`self` 下「员工线索 TOP10」改「我的线索状态分布」。

**验证（服务器）**：管理员 `scope=all`，新建 2 条线索 → `totalLeads=2` ✅；无角色用户 `scope=self` → `totalLeads=1`（仅含分配给自己的线索）✅

### 修复三：顶栏右上角显示当前登录用户姓名（不再写死「管理员」）— 提交 `e6b7dec`
**问题**：无论谁登录，顶栏右上角始终显示「管理员」，不显示登录者本人姓名。

**根因**：`src/components/Header.tsx` 原第 250 行将用户名写死为 `<span className="text-sm text-gray-700">管理员</span>`，未读取当前登录用户信息。

**改动**：Header 新增状态 `currentUser` 与 `useEffect`，调用 `/api/auth/me`（该接口依赖中间件注入的 `x-user-id` 解析当前用户，返回 `realName/username/roles`）；右上角改为显示 `realName || username`（兜底「用户」），并在姓名下方展示首个角色名。

**验证（服务器）**：`docker compose build` 输出 `BUILD_DONE`，容器 `xinchen-erp` 已启动；部署后源码已无写死「管理员」，动态显示代码就位（line 267）。登录任意非管理员账号，右上角即显示其本人姓名与角色。

### 修复四：员工账号自动化（手机号登录 / 入职自动建号 / 离职停用 / 数据重分配）
**需求**：①员工登录默认使用手机号（而非员工 id）；②今后新员工入职自动建立系统账号；③员工离职自动取消登录权限，但其业务数据由管理员重新分配（不丢失）。

**改动清单**：

| 文件 | 改动 |
|------|------|
| `src/app/api/auth/login/route.ts` | 登录匹配由 `findUnique({username})` 改为 `findFirst({ where: { OR: [{username}, {phone: username}] } })`，支持手机号或用户名登录 |
| `src/lib/employee-account.ts`（新增） | `resolveAccountUsername()` 助手：显式 username 优先，否则默认手机号；用户名冲突时追加数字后缀保证唯一（登录仍可用手机号匹配） |
| `src/app/api/employees/route.ts`（POST） | 员工有手机号且尚未关联账号时 **自动建立系统账号**（username=手机号，默认密码 `Xc@123456`，强制改密，按 `status` 设 `isActive`） |
| `src/app/api/employees/[id]/route.ts`（PUT） | 编辑时若无账号自动补建；**状态变更时同步 `user.isActive`**（inactive→`false` 取消登录权限，active→`true` 恢复） |
| `src/app/api/employees/reassign/route.ts`（新增） | GET 统计离职员工名下待分配数据（`Lead`/`Student`/`Order`/`CopywriterTask` 的 `assignedToId` 计数）；POST 将上述 `assignedToId` 从 `fromUserId` 转移到 `toUserId` |
| `src/app/login/page.tsx` | 账号框标签改为「手机号 / 用户名」，placeholder 提示可用手机号 |
| `src/app/(main)/employees/page.tsx` | 表单「登录账号」提示改为「默认手机号」；离职员工行新增「重新分配数据」按钮（UserPlus 图标），弹窗展示待分配数据数量并选择在职目标员工一键转移 |

**效果**：新员工（填手机号）保存即自动拥有登录账号，可用手机号登录；将员工置为「离职」后其账号自动停用（登录被拒），但线索/学生/订单等数据保留，管理员在员工列表点「重新分配数据」即可批量转交给在职同事。
