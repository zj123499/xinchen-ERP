## 一、修复一：账号和密码设置逻辑优化（避免误解）

### 1.1 问题
原「员工管理」中创建/编辑账号时，密码输入框语义不清：留空=默认密码、填写=自定义密码，但界面无任何提示，管理员容易误以为「填写了却没生效」或「忘记改密码员工一直用默认密码登录」，造成沟通与安全隐患。

### 1.2 根因
- 后端 `password` 字段留空即静默使用默认密码 `Xc@123456`，无显式标识。
- 前端只有单个密码框，无「确认密码」「显示密码」「当前状态」提示。
- 列表页无法区分「默认密码 / 自定义密码 / 待改密」，管理员无法快速识别风险账号。

### 1.3 改动清单
| 文件 | 改动 |
|------|------|
| `prisma/schema.prisma` | `User` 新增 `isDefaultPassword Boolean @default(true)`（映射 `is_default_password`），标识当前是否仍用默认密码 |
| `src/app/api/employees/route.ts` | POST 解析 `confirmPassword`，`password !== confirmPassword` 返回 400；创建时 `isDefaultPassword: !password`（填了=自定义 false，留空=默认 true） |
| `src/app/api/employees/[id]/route.ts` | PUT 同样校验两次密码一致；重置/编辑维护 `isDefaultPassword` |
| `src/app/(main)/employees/page.tsx` | 新增「显示密码」开关、二次「确认密码」输入、动态提示文案（填了→"将以自定义密码创建，请牢记"；留空→"留空则使用默认密码，首次登录需修改"）；列表「登录账号」列新增徽标：默认密码=琥珀、自定义=绿、待改密=蓝 |
| `prisma/seed.ts` | admin 与管理员示例账号标记 `isDefaultPassword: false` |
| `src/app/api/auth/change-password/route.ts` | 自助改密成功后置 `isDefaultPassword: false` |

### 1.4 验证（服务器）
- 自定义密码创建 → 登录 200 ✅
- 两次密码不一致 → 创建拦截 400 ✅
- 重置为默认密码 → 返回 `isDefaultPassword: true` 200 ✅

## 二、修复二：工作台按个人权限展示（不再全员看全公司）

### 2.1 问题
工作台（`/`）原本对所有登录人都展示全公司汇总数据（线索总量、合同总额、回款等），与「按权限隔离数据」的设计冲突，存在信息越权风险。

### 2.2 根因（排查链路）
1. **残留 docker build 冲突**：上一次 `docker compose build` 未结束，新构建被 CANCELED，导致部署停留在旧镜像。→ `pkill -9 -f "docker compose build"` 后重部署。
2. **错误导入鉴权函数**：曾误从 `@/lib/auth` 导入 `verifyToken/getTokenFromRequest`，该文件仅导出 `hashPassword/verifyPassword`，引发 TS 类型错误。→ 移除该依赖。
3. **中间件注入的 `x-tenant-id` 头恒为 0**（排查关键）：dashboard 依赖中间件注入的 `x-tenant-id`/`x-user-id` 头，实测 `x-user-id=1`、`x-user-roles=admin` 正常，但 `tenantId=0`，导致所有 `where: { tenantId: 0 }` 计数恒为 0。→ 改为**直接从 JWT payload 解析** `tenantId / userId / roles`。
4. **Dockerfile 吞错**：原 `RUN npx next build 2>&1 | tail -20` 把构建失败的错误码吞掉，容器误判构建成功并停留在旧版。→ 改为 `RUN npx next build`，并在注释中标注「不要用 `| tail` 吞掉错误码」。

### 2.3 最终方案
`src/app/api/dashboard/route.ts` 改为：
- `import { verifyToken } from "@/lib/jwt";`
- `getToken(request)` 从 `authorization` Bearer 或 cookie `token` 提取
- `const payload = await verifyToken(token)` → 取 `tenantId / userId / roles`
- 数据范围：`ALL_SCOPE_ROLES = ["admin","manager","finance"]`，命中则 `scope="all"`（全公司），否则 `scope="self"`（仅自己负责）
- 数据归属过滤：Lead/Student/Order → `assignedToId`；Contract/Payment/Application → `student.assignedToId`；Commission → `employee.userId`；VisitRecord → `visitorId`；成本仅全量角色可见
- 响应体新增 `scope` 字段

前端 `src/app/(main)/page.tsx`：
- 标题区加徽标：`我的工作台（仅看我负责的数据）` / `全公司看板`
- `self` 下「员工线索 TOP10」卡片标题改为「我的线索状态分布」

### 2.4 验证（服务器）
- 管理员 `scope=all`，新建 2 条线索 → `totalLeads=2` ✅
- 无角色用户 `scope=self` → `totalLeads=1`（仅含分配给自己的线索）✅

## 三、部署流程与已知坑（经验沉淀）

### 3.1 标准部署流水线
本地改码 → `deploy_batch.js` 打包上传 → `docker compose build --no-cache` → `up` → entrypoint 自动 `prisma db push` + seed。

### 3.2 已记录坑与对策
| 坑 | 现象 | 对策 |
|----|------|------|
| 残留构建进程 | 新构建 CANCELED | 部署前先 `pkill -9 -f "docker compose build"` |
| 鉴权函数误导入 | TS 类型错误 | 只用 `@/lib/jwt` 的 `verifyToken`，勿用 `@/lib/auth` |
| 中间件 tenantId=0 | 统计计数全 0 | 直接从 JWT payload 解析 tenantId/userId/roles |
| Dockerfile `| tail` 吞错 | 构建失败却判成功、停留旧版 | `RUN npx next build` 不加管道 |
| seed 缺 bcryptjs | 新环境首部署播种失败 | prisma-builder 阶段补装 bcryptjs（已知项，待修） |
